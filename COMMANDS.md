## üõ†Ô∏è Development Commands

### `/by-ai:commit-suggest`

**Description**: Suggests a well-formatted commit message with conventional commit standards based on current changes without executing the commit.

**Usage**:

```bash
/by-ai:commit-suggest
/by-ai:commit-suggest --breaking
/by-ai:commit-suggest --type <type>
/by-ai:commit-suggest --ticket <ticket>
```

**Options**:

- `--breaking`: Force breaking change structure
- `--type <type>`: Force specific commit type (feat, fix, docs, style, refactor, perf, test, chore) (if not provided, infers from changes)
- `--ticket <ticket>`: Force specific ticket ID (if not provided, extracts from branch name or prompts user)

**What it does**:

1. Extracts ticket ID from branch name or uses provided ticket
2. Analyzes staged changes with `git diff --cached` (if no staged files, runs `git add .`)
3. Suggests conventional commit message format: `[<ticket>] <type>: <description>`
4. Copies the complete `git commit -m "<message>"` command to clipboard

---

### `/by-ai:summarize-changes`

**Description**: Quickly summarizes current changes using git diff for commit message preparation.

**Usage**:

```bash
/by-ai:summarize-changes
/by-ai:summarize-changes --short
```

**Options**:

- `--short`: Generate a single-line summary instead of bullet points

**What it does**:

1. Analyzes all local changes with `git diff HEAD`
2. Identifies file types modified and nature of changes
3. Outputs structured summary in commit message style
4. Maximum 72 characters per line, present tense, imperative mood

## üß™ Testing Commands

### `/by-ai:write-unit-tests`

**Description**: Generate unit and integration tests for newly added code or specified existing files.

**Usage**:

```bash
/by-ai:write-unit-tests
/by-ai:write-unit-tests [file_to_test]
```

**Parameters**:

- `[file_to_test]`: Optional - specific file to create tests for

**What it does**:

1. **Framework Detection**: Identifies testing framework (Jest, Vitest, Mocha, PyTest, etc.)
2. **Code Analysis**: Analyzes new code or specified file
3. **Test Planning**: Determines unit vs integration test needs
4. **Implementation**: Creates comprehensive tests following AAA pattern
5. **Coverage**: Tests happy path, edge cases, and error conditions

**Features**:

- Follows project-specific testing conventions
- Creates proper mocks and stubs
- Implements security and validation testing
- Runs linter/formatter on generated tests
- **Important**: Never modifies implementation code, only creates test files

---

### `/by-ai:manual-test`

**Description**: Manual testing of production changes using Chrome browser automation with Playwright MCP.

**Usage**:

```bash
/by-ai:manual-test [production_url] [commit_hash]
/by-ai:manual-test [production_url] [commit_hash] --no-verify
```

**Parameters**:

- `[production_url]`: Production website URL to test
- `[commit_hash]`: Specific commit to analyze for testing scope
- `--no-verify`: Skip manual test plan confirmation

**What it does**:

1. **Change Analysis**: Uses `git diff` to analyze commit changes
2. **Test Planning**: Creates structured manual test plan
3. **Browser Testing**: Opens production URL and executes manual tests
4. **Results**: Reports issues, bugs, UX concerns, or test confirmations

**Requirements**: Playwright MCP server must be configured

---

### `/by-ai:e2e-test:plan`

**Description**: Explore website and plan end-to-end test scenarios for later implementation.

**Usage**:

```bash
/by-ai:e2e-test:plan [website_url]
```

**What it does**:

1. Explores target website functionality using specialized subagent
2. Gathers comprehensive information about website features
3. Creates detailed e2e test plan in `@test_plan.md` file

**Requirements**: Playwright MCP server for website exploration

---

### `/by-ai:e2e-test:write`

**Description**: Create Playwright e2e test scenarios based on existing test plan in `@test_plan.md` generated by `/by-ai:e2e-test:plan`.

**Usage**:

```bash
/by-ai:e2e-test:write
```

It is strongly recommended to run `/by-ai:e2e-test:write` with prompting a specific part of the test plan to implement:

```bash
/by-ai:e2e-test:write [test_plan_section]
```

**What it does**:

1. Reads existing `@test_plan.md` from project root
2. Creates Playwright test scenarios for unmarked items
3. Updates test plan by marking completed items with checkmarks

**Requirements**:

- `@test_plan.md` file must exist in project root
- Playwright MCP server configuration

## üîç Code Analysis Commands

### `/by-ai:code-review`

**Description**: Comprehensive code analysis across quality, security, performance, and architecture domains.

**Usage**:

```bash
/by-ai:code-review [branch]
```

**Parameters**:

- `[branch]`: Target branch for comparison (defaults to 'master')

**What it does**:

1. **File Discovery**: Uses `git diff {branch}...$(git branch --show-current)` to identify changed files
2. **Multi-Domain Analysis**:

   - **Quality**: Code style, complexity, maintainability, SOLID principles
   - **Security**: Vulnerability scanning, authentication flaws, input validation
   - **Performance**: Bottleneck identification, memory management, scalability
   - **Architecture**: Design patterns, modularity, coupling analysis
   - **Testing**: Test coverage and quality analysis (without running tests)
   - **Maintainability**: Code readability, documentation quality

3. **Severity Classification**:

   - **High**: Critical issues requiring immediate attention
   - **Medium**: Important issues to resolve soon
   - **Low**: Minor issues for later resolution

4. **Output Format**: Compact markdown table with:
   - Assessment summary with checkmarks for good areas
   - Detailed improvements table with file paths, issues, severity, and recommendations
   - File paths in format: `./path/to/file.js:line_number`

## üìú Utility Scripts

### `scripts/commit.sh`

**Description**: Interactive script for creating AI-assisted git commits with structured format.

**Usage**:

```bash
./scripts/commit.sh [--breaking] [--type <type>] [--ticket <ticket>] [-s|--short] [-h|--help]
```

**Options**:

- `--breaking`: Mark commit as breaking change
- `--type <type>`: Specify commit type (feat, fix, docs, style, refactor, perf, test, chore)
- `--ticket <ticket>`: Specify ticket number manually
- `-s, --short`: Generate short summary instead of detailed bullet points
- `-h, --help`: Show help message

**Process**:

1. Starts AI summary generation in background
2. Extracts ticket number from branch name or prompts for input
3. Interactive commit type selection
4. User confirmation/editing of generated commit message
5. Executes `git commit` with finalized message

---

### `scripts/find-ticket-from-branch.sh`

**Description**: Extracts ticket numbers from git branch names using pattern matching.

**Usage**:

```bash
./scripts/find-ticket-from-branch.sh [branch-name]
```

**Parameters**:

- `[branch-name]`: Optional branch name (defaults to current branch)

**Supported Patterns**:

- Standard format: `ABC-1234`, `PROJ-123`
- Pattern: `[A-Z]+-[0-9]+`
- Fallback: Numbers only (converted to `TICKET-{number}`)

---

### `scripts/session-init.sh`

**Description**: Claude Code session initialization with configuration validation and update checks.

**What it checks**:

1. **CLAUDE.md**: Verifies existence and proper submodule referencing
2. **MCP Configuration**: Validates Playwright MCP server setup
3. **Git Submodule**: Checks for configuration updates
4. **Update Notifications**: Shows available configuration updates

**Automatic Execution**: Runs automatically when starting Claude Code sessions

---

### `scripts/init-submodule.sh`

**Description**: Initializes Byborg AI configuration when used as a git submodule.

**Usage**:

```bash
./scripts/init-submodule.sh [target_directory] [submodule_path]
```

**Parameters**:

- `[target_directory]`: Directory to create files in (default: current directory)
- `[submodule_path]`: Path to submodule (default: .claude)

**Creates**:

- `.mcp.json`: Playwright MCP server configuration
- `CLAUDE.md`: Reference to submodule configuration

---

### `scripts/check-mcp-for-testing.sh`

**Description**: Validates MCP server availability before running testing commands.

**Usage**:

```bash
./scripts/check-mcp-for-testing.sh
```

**Validation**: Uses `claude mcp get playwright` to verify server status and provides setup instructions if not configured.

---

### `scripts/install-pre-commit-hook.sh`

**Description**: Installs pre-commit hook that automatically runs code review before commits.

**Usage**:

```bash
./scripts/install-pre-commit-hook.sh
```

**Features**:

- Interactive branch selection for code review comparisons
- Automatic backup of existing pre-commit hooks
- Submodule-aware installation
- User confirmation prompt after code review

**Hook Behavior**:

1. Runs `/by-ai:code-review` on staged changes
2. Displays review results
3. Prompts user for commit confirmation
4. Supports `git commit --no-verify` to bypass

## üîß MCP Server Requirements

Several commands require the **Playwright MCP server** for browser automation:

- `/by-ai:manual-test`
- `/by-ai:e2e-test:plan`
- `/by-ai:e2e-test:write`

**Setup Commands**:

```bash
# Project-scoped installation
claude mcp add playwright npx @playwright/mcp@latest --scope project

# Global installation
claude mcp add playwright npx @playwright/mcp@latest --scope global
```
